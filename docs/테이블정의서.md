# QA-Server 테이블 정의서

- **DB**: PostgreSQL
- **연결 정보**: `postgresql://qa_user:qa_pass@localhost:5432/qa_db`
- **ORM**: SQLAlchemy (`backend/models.py`)
- **마이그레이션**: Alembic (`backend/alembic/`) — `alembic upgrade head`로 반영
- **최종 수정**: 2026-07-10 (project_presets · encryption_configs 추가)

---

## 1. projects (프로젝트)

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| name | VARCHAR(255) | N | — | 프로젝트명 |
| created_at | TIMESTAMP(TZ) | Y | now() | 생성일시 |
| updated_at | TIMESTAMP(TZ) | Y | — | 수정일시 (on update) |

**관계**: 1:N → qa_snapshots, deploy_histories, test_runs, test_flows, notification_configs, case_histories, test_suites, project_presets, encryption_configs

## 2. qa_snapshots (QA 데이터 스냅샷)

케이스 관리 / 테스트결과서 / 배포결과서 3탭의 전체 상태를 프로젝트 단위로 JSON 통째 저장. 프로젝트당 1행 (upsert).

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| data | JSON | N | — | 전체 QA 데이터 (아래 구조 참고) |
| updated_at | TIMESTAMP(TZ) | Y | now() | 갱신일시 (on update) |

### data JSON 구조

```jsonc
{
  "mgr": {
    "cats": [{ "id": "mc1700000000", "name": "로그인", "color": "#4f8cff" }],
    "cases": [{
      "id": "TC-001", "name": "로그인 성공", "type": "Positive",
      "method": "POST", "endpoint": "/auth/login", "expectedStatus": 200,
      "pf": "Pass", "owner": "홍길동", "date": "2026-06-17T10:00",
      "catId": "mc1700000000", "headers": [], "queryParams": [], "body": "",
      "assertions": [{ "target": "body_json", "path": "data.status", "operator": "eq", "value": "APPROVED" }],
      "encrypted": false, "encryptionScope": "body", "encryptionKeyBase64": "", "encryptionMode": "GCM", "encryptedFieldKeys": []
    }]
  },
  "tst": {
    "cover": { "company":"","project":"","version":"v1.0","environment":"","author":"","approver":"","date":"","start":"","end":"" },
    "cats": [],
    "cases": [{ "id":"TC-001","catId":"mc1700000000","actual":"","pf":"Pass","owner":"","date":"","notes":"","evidence":[] }]
  },
  "dep": {
    "cover": { "company":"","project":"","version":"","environment":"Production","deployType":"정기 배포","deployer":"","date":"","target":"","summary":"" },
    "cats": [],
    "cases": [{ "id":"TC-001","catId":"mc1700000000","actual":"","pf":"완료","owner":"","date":"","notes":"","evidence":[] }]
  }
}
```

> 케이스 마스터 정보(이름/입력/기대결과)는 `mgr.cases`만 가지고, `tst` / `dep`는 동일 `id`로 매핑되는 실행 결과만 저장.

## 3. deploy_histories (배포 이력 — LLM 분석용)

`qa_snapshots.data.dep` 저장 시 핵심 필드만 뽑아 정규화 동기화.

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| version | VARCHAR(100) | Y | — | 배포 버전 (예: v2.3.1) |
| environment | VARCHAR(50) | Y | — | Production / Staging / Development |
| deploy_type | VARCHAR(100) | Y | — | 정기 배포 / 긴급 배포 / 신규 기능 / 패치 / 롤백 |
| deployer | VARCHAR(100) | Y | — | 배포자 |
| target_server | VARCHAR(255) | Y | — | 대상 서버 |
| summary | TEXT | Y | — | 배포 내용 요약 |
| total_cases | INTEGER | Y | 0 | 점검 항목 전체 건수 |
| done_cases | INTEGER | Y | 0 | 완료 건수 |
| fail_cases | INTEGER | Y | 0 | 미완료 건수 |
| deployed_at | TIMESTAMP(TZ) | Y | — | 배포 완료 시각 |
| created_at | TIMESTAMP(TZ) | Y | now() | 레코드 생성일시 |

**Upsert 기준**: `project_id + version + environment` 조합이 같으면 갱신, 없으면 삽입.

## 4. test_flows (테스트 플로우)

업무 단위로 순서가 고정된 케이스 묶음. 한 스텝 실패 시 이후 스텝 건너뜀 (stop-on-fail).

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| name | VARCHAR(255) | N | — | 플로우명 |
| steps | JSON | N | [] | `[{"case_id":"TC-001","order":0}, ...]` |
| created_at | TIMESTAMP(TZ) | Y | now() | 생성일시 |
| updated_at | TIMESTAMP(TZ) | Y | — | 수정일시 (on update) |

## 5. test_runs (자동 실행 작업)

자동 실행 1회 = 1행. 완료 후 결과 컬럼은 **불변(immutable)**.

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| base_url | VARCHAR(500) | Y | — | 실행 대상 서버 URL |
| status | VARCHAR(20) | Y | pending | pending / running / done / failed |
| total | INTEGER | Y | 0 | 총 실행 케이스 수 (개별 + 플로우 스텝 합계) |
| done | INTEGER | Y | 0 | 완료된 케이스 수 |
| fail | INTEGER | Y | 0 | 실패 케이스 수 |
| error | TEXT | Y | — | 실행 오류 메시지 |
| label | VARCHAR(255) | Y | — | 사용자 지정 레이블 (유일하게 수정 가능) |
| case_ids | JSON | Y | — | 개별 실행 케이스 ID 목록 `["TC-001", ...]` |
| flow_ids | JSON | Y | — | 실행한 플로우 ID 목록 `[1, 2]` |
| case_results | JSON | Y | — | 케이스별 결과 스냅샷 `[{case_id, pf, actual, notes}]` |
| flow_results | JSON | Y | — | 플로우별 결과 스냅샷 |
| mgr_snapshot | JSON | Y | — | 실행 시점 케이스 내용 `[{id, name, endpoint, method, catId}]` |
| started_at | TIMESTAMP(TZ) | Y | now() | 실행 시작일시 |
| finished_at | TIMESTAMP(TZ) | Y | — | 실행 완료일시 |

### flow_results JSON 구조

```jsonc
[{
  "flow_id": 1, "flow_name": "결제 생성 플로우", "status": "done",
  "steps": [{ "case_id": "TC-002", "pf": "Pass", "skipped": false }]
}]
```

**관계**: N:1 → projects, 1:N → run_comments

## 7. notification_configs (알림 설정)

프로젝트별 Discord / Slack 웹훅 알림 설정. 실행 완료/실패 시 자동 전송.

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| name | VARCHAR(100) | N | — | 설정 식별용 이름 |
| type | VARCHAR(20) | N | — | `discord` \| `slack` |
| webhook_url | VARCHAR(500) | N | — | 웹훅 수신 URL |
| enabled | BOOLEAN | Y | true | 활성화 여부 |
| events | JSON | Y | [] | 구독 이벤트 목록 `["run_completed", "run_failed"]` |
| attach_excel | BOOLEAN | Y | false | Discord 알림 시 테스트 수행 내역서 Excel 첨부 여부 |
| created_at | TIMESTAMP(TZ) | Y | now() | 생성일시 |

**지원 이벤트**

| 이벤트 | 설명 |
|---|---|
| `run_completed` | 실행 정상 완료 (fail 포함 가능) |
| `run_failed` | 실행 자체가 오류로 종료 |

## 8. case_histories (케이스 변경 이력)

케이스 관리 저장 시 이전 스냅샷과 diff 비교 후 자동 기록. 케이스 스펙(입력·기대값) 변경만 추적.

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| case_id | VARCHAR(100) | N | — | 케이스 ID (index) |
| action | VARCHAR(20) | N | — | `created` \| `updated` \| `deleted` |
| before | JSON | Y | — | 변경 전 케이스 전체 (updated/deleted 시 존재) |
| after | JSON | Y | — | 변경 후 케이스 전체 (created/updated 시 존재) |
| changed_at | TIMESTAMP(TZ) | Y | now() | 변경 감지 일시 |

**추적 필드**: `name`, `method`, `endpoint`, `expectedStatus`, `catId`, `type`, `headers`, `queryParams`, `body`

> `pf`, `actual`, `notes`, `owner`, `date` 등 실행 결과 필드는 추적하지 않음.

## 9. test_suites (테스트 스위트)

프로젝트의 비즈니스 로직 단위를 케이스 + 플로우 묶음으로 저장. 프로젝트 기본 스위트(`is_default`) 설정 시 진입 시 자동 적용.

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| name | VARCHAR(255) | N | — | 스위트 이름 |
| description | TEXT | Y | — | 설명 |
| case_ids | JSON | Y | [] | 포함된 케이스 ID 목록 `["TC-001", ...]` |
| flow_ids | JSON | Y | [] | 포함된 플로우 ID 목록 `[1, 2]` |
| is_default | BOOLEAN | Y | false | 프로젝트 기본 스위트 여부 (프로젝트당 1개만 true) |
| created_at | TIMESTAMP(TZ) | Y | now() | 생성일시 |
| updated_at | TIMESTAMP(TZ) | Y | — | 수정일시 (on update) |

**동작 규칙**
- `is_default=true` 설정 시 같은 프로젝트의 기존 기본 스위트는 자동 해제
- 스위트 "적용" 시 Zustand `activeSuiteId`에 저장 → 케이스 관리 / 자동 실행 양쪽에 즉시 반영
- 프로젝트 변경 시 `activeSuiteId` 자동 초기화

## 10. project_presets (저장된 값 / 프리셋)

케이스 등록 시 반복 입력되는 헤더·URL·경로·파라미터·바디 필드·판정조건 경로를 프로젝트 단위로 등록해두고 재사용.

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| kind | VARCHAR(20) | N | — | `header` \| `url` \| `param` \| `path` \| `body` \| `assertion_path` |
| label | VARCHAR(100) | N | — | 식별용 이름 (예: "인증 토큰") |
| key | VARCHAR(200) | Y | — | header/param/body의 키 (url/path/assertion_path는 사용 안 함) |
| value | TEXT | N | — | 실제 값 |
| category_id | VARCHAR(50) | Y | — | 지정 시 케이스 등록 화면에서 해당 카테고리 선택만으로 자동 적용됨 (`qa_snapshots.data.mgr.cats[].id`, FK 아님 — QA 데이터가 JSON으로 별도 저장되기 때문) |
| created_at | TIMESTAMP(TZ) | Y | now() | 생성일시 |

**동작 규칙**
- `category_id`가 null이면 "카테고리 공통" — 자동 적용 안 되고 항상 수동 선택
- 케이스 편집 폼에서 카테고리를 바꾸면, 이전 카테고리가 자동 적용해뒀던 헤더/파라미터/바디 키만 골라 제거하고 새 카테고리 값으로 교체 (수동으로 추가한 값은 보존)

## 11. encryption_configs (암호화 설정)

프로젝트별 AES-256-GCM 암호화 키. 헤더/URL 등을 다루는 `project_presets`와 별도 테이블·화면으로 분리 (추후 사용자/관리자 권한으로 메뉴 노출을 나눌 수 있도록 의도적으로 구분).

| 컬럼명 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| id | INTEGER | N | auto increment | PK |
| project_id | INTEGER | N | — | FK → projects.id |
| label | VARCHAR(100) | N | — | 식별용 이름 (예: "결제 암호화 키") |
| mode | VARCHAR(20) | N | GCM | 현재는 GCM만 지원 |
| key_base64 | VARCHAR(100) | N | — | AES-256 키 32바이트를 Base64로 인코딩한 값 |
| created_at | TIMESTAMP(TZ) | Y | now() | 생성일시 |

**동작 규칙**
- 등록 화면의 `+ 추가`는 키를 직접 입력받지 않고 Web Crypto로 안전한 랜덤 32바이트 키를 생성해 Base64로 저장
- IV(초기화 벡터, 12바이트)는 이 테이블에 저장하지 않고 케이스 실행마다 새로 무작위 생성 — 바디 전체 암호화 시 암호문 앞에 붙여 같이 전송, 필드별 암호화 시 `X-IV` 헤더로 별도 전송
- 케이스(`qa_snapshots.data.mgr.cases[].encryptionKeyBase64`)에 이 테이블의 `key_base64` 값을 그대로 복사해서 저장 (FK 관계 아님 — 케이스가 삭제된 설정을 참조해도 실행에는 영향 없도록 값 자체를 스냅샷)

## 불변성 정책

| 테이블 | 수정 가능 | 불변 |
|---|---|---|
| test_runs | label | case_ids, flow_ids, case_results, flow_results, mgr_snapshot |

| case_histories | (없음) | 모든 컬럼 (자동 기록, 수정·삭제 엔드포인트 없음) |
