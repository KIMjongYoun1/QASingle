# Single_QA_Tools — 서비스 소개서

---

## 개요

**Single_QA_Tools**는 소규모 개발팀 또는 개인 QA 담당자가 테스트 케이스 관리부터 자동화 실행, 분석, 배포 점검까지 하나의 인터페이스에서 수행할 수 있는 통합 QA 관리 도구입니다.

---

## 핵심 기능

### 0. 전체 대시보드 (메인 페이지)

- 앱 진입 시 모든 프로젝트 현황을 카드 형태로 표시
- **최근 실행일 기준 정렬** — 활성 프로젝트가 상단
- 카드당 실행 횟수 / Pass율 / 변경 이력 건수 + 통과율 바 표시
- 카드 클릭 → 해당 프로젝트 히스토리 탭으로 즉시 이동
- **기간 필터**: 전체 / 7일 / 30일 / 90일 / 직접 설정 — 모든 카드·차트에 일괄 적용
- 프로젝트 수 무제한 대응: 초기 6개 노출, **더보기** 버튼으로 6개씩 추가
- 2개 이상 프로젝트 시 비교 차트 자동 표시
  - 프로젝트별 Pass율 바차트
  - 프로젝트별 실행 횟수 바차트
  - 케이스별 Pass율 레이더차트 (공통 실행 케이스 기준)

### 1. Test Suite (스위트 관리)

- 프로젝트 단위 비즈니스 로직 묶음을 **스위트**로 명명하여 저장
- 스위트 = 케이스 ID 목록 + 플로우 ID 목록의 조합
- 프로젝트당 **기본 스위트(is_default)** 1개 지정 가능
- **Additive(합산) 방식**: 스위트 "불러오기" 클릭 시 기존 선택에 추가 (덮어쓰기 아님)
  - 여러 스위트를 동시에 불러올 수 있으며, 히스토리와 스위트를 함께 조합 가능
  - 케이스 관리 탭: 불러온 스위트·히스토리의 케이스 ID를 합산(union)하여 필터 표시
  - 자동 실행 탭: 스위트 케이스 + 플로우 자동 프리셀렉트
  - 칩(chip) 형태로 각 소스를 개별 해제하거나 전체 해제 가능
- 스위트 화면에서 케이스를 **직접 추가·수정·삭제** 가능 — 케이스 관리 탭과 실시간 동기화
  - "+ 케이스 추가" 버튼 → 폼 다이얼로그 → 저장 시 mgr.cases 등록 + 현재 스위트에 자동 포함
  - 케이스 행 hover 시 수정(✏️) · 삭제(🗑️) 버튼 표시
- 스위트 CRUD: 이름/설명 편집, 케이스 체크박스 토글, 플로우 체크박스 토글
- 기본 스위트 설정 (`★`) / 해제
- 체크박스 토글 즉시 DB 반영 (별도 저장 버튼 없음)

### 2. 케이스 관리 (Case Manager)

- 테스트 케이스를 카테고리 단위로 구조화 관리
- 각 케이스에 **Postman 표준 방식**으로 실행 파라미터 직접 입력
  - 케이스별 Base URL (비워두면 전역 URL fallback)
  - HTTP 메서드 / 엔드포인트 / 기대 상태코드
  - Headers / Query Params / Body (고급 설정)
- **성공 판정 조건(Assertions)** — QA 담당자가 웹 UI에서 직접 설정
  - 대상: 상태코드 / Body JSON 경로(`data.success`) / Body 텍스트 / 헤더
  - 연산자: `=` / 포함 / 미포함 / 존재 / 미존재 / `>` / `<`
  - 모든 조건이 AND로 충족될 때만 Pass 처리
- Pass / Fail / 보류 상태 수동 기록
- 연동규격서 엑셀 포맷 임포트 지원
- **복수 소스 불러오기**: 스위트 여러 개 + 히스토리를 additive union으로 동시 조합
  - 각 소스는 칩 형태로 표시, 개별 해제 가능
- 히스토리에서 이전 실행 설정 불러오기 — 케이스 필터 원클릭 복원

### 3. 자동 실행 (Auto Run)

- 선택된 개별 케이스 + 테스트 플로우를 순차·병렬 혼합 실행
- 실제 HTTP 요청을 서버가 대신 수행 (Human-in-the-loop 방식)
- **전역 설정 + 케이스별 개별 설정 공존**
  - **전역 Base URL**: 실행 버튼 영역에서 입력, 프로젝트별 저장
  - **전역 기본 헤더**: KV 테이블 형태로 Authorization 등 여러 헤더 등록, 프로젝트별 저장
  - **케이스별 Base URL**: 개별 케이스에 URL 지정 시 전역 URL 대신 사용 (override)
  - **케이스별 헤더**: 개별 케이스 헤더가 전역 헤더보다 우선 적용 (merge & override)
- 실행 결과(actual / pf / notes)를 케이스 단위로 기록, 이후 불변 저장
- 히스토리 불러오기로 이전 실행 케이스 선택 즉시 복원
- 실행 완료 / 실패 시 Discord · Slack 알림 자동 전송 (설정된 경우)

### 4. 테스트 플로우 (Test Flow)

- 업무 흐름 단위로 케이스를 순서대로 묶은 플로우 정의
- 스텝 실패 시 이후 스텝 자동 건너뜀 (stop-on-fail)
- 플로우 단위 실행 결과 별도 저장

### 5. 실행 히스토리 (History)

- 모든 자동 실행 이력을 시계열로 보관
- 실행 단위 레이블 편집 가능 (나머지 결과는 불변)
- 실행별 상세: 개별 케이스 결과 + 플로우 스텝 결과
  - Pass / Fail 필터, 카테고리 필터 지원
  - 각 케이스 행 클릭 시 actual / notes 인라인 확인
- 실행 단위 댓글 추가 (append-only)
- 왼쪽 패널: 실행 목록 검색 + 전통과 / 실패포함 필터
- 히스토리 설정 → 케이스관리 / 자동실행 양방향 복원

### 6. 실행 분석 (Analytics)

- 프로젝트별 실행 분석 전용 탭
- **기간 필터** 지원 (전체 / 7일 / 30일 / 90일 / 직접 설정)
- 요약 지표: 전체 실행 횟수 / 전체 통과율 / 분석 케이스 수
- Pass율 추이 라인 차트 (최근 30회)
- 실행별 Pass / Fail 스택 바차트
- 케이스별 실패 횟수 순위 — 수평 바차트 + 상세 테이블

### 7. 케이스 변경 이력 (Case History)

- 케이스 관리 저장 시 이전 스냅샷과 자동 diff 비교 후 기록
- **생성**: 추가된 케이스의 전체 내용 표시
- **수정**: 변경된 필드만 변경 전 → 변경 후 나란히 표시
- **삭제**: 삭제된 케이스의 전체 내용 표시
- 케이스 ID 검색 + 생성 / 수정 / 삭제 필터 (건수 함께 표시)
- 추적 필드: 이름, 메서드, 엔드포인트, 기대 상태코드, 카테고리, 유형, 헤더, 쿼리 파라미터, 바디

### 8. 알림 설정 (Notifications)

- 프로젝트별 Discord / Slack 웹훅 복수 등록
- 구독 이벤트 선택: `run_completed` (실행 완료) / `run_failed` (실행 오류)
- 테스트 전송 버튼으로 웹훅 즉시 검증
- 활성화 / 비활성화 토글
- **알림 메시지 구성**
  - 요약 카드: 프로젝트명 · 실행 레이블 · 대상 서버 · 전체/통과/실패 수
  - 실패 케이스 목록 (최대 10건, 초과 시 "외 N건" 표기)
  - 테스트 플로우 결과 (플로우명 + ✅/❌)
  - 문서 형식 요약 보고서 (code block) — 통과율 · 일시 · 서버 포함
- 알림 설정 페이지 내 **메시지 미리보기** — Discord/Slack 렌더링 탭 전환으로 확인

### 9. 테스트결과서 (TST)

- 최종 테스트 결과를 문서 형식으로 정리
- 커버 페이지(프로젝트명 / 환경 / 작성자 / 승인자) + 케이스별 결과 기록
- 사이드바 케이스 트리로 특정 케이스 바로 이동

### 10. 배포결과서 (DEP)

- 배포 점검 결과 기록 및 이력 관리
- 배포 버전 / 환경 / 배포 유형(정기 / 긴급 / 롤백 등) 분류
- LLM AI 분석: 배포 이력을 기반으로 패턴 및 위험 포인트 분석

  ├── /api/qa                QA 스냅샷 load/save + 케이스 변경 이력 조회
  ├── /api/runs              자동 실행 생성 · 조회 · 폴링
  ├── /api/flows             테스트 플로우 CRUD
  ├── /api/analytics         실행 분석 (기간 필터 지원)
  ├── /api/notifications     알림 설정 CRUD + 테스트 전송
  ├── /api/suites            Test Suite CRUD + 기본 스위트 설정
  ├── /api/deploy            배포결과서 이력
  └── /api/analysis          LLM 분석 (Claude API)
        │
        ▼
PostgreSQL (9개 테이블)
  projects · qa_snapshots · deploy_histories · test_flows
  test_runs · run_comments · notification_configs · case_histories · test_suites

  app.log            ← 전체 요청 (일별 로테이션)

  runs.log           ← 자동 실행 전용

## 사용 시나리오

1. **첫 진입**: 대시보드에서 전체 프로젝트 현황 확인 → 카드 클릭으로 프로젝트 진입
2. **신규 프로젝트 셋업**: 프로젝트 생성 → 카테고리 정의 → 케이스 입력 (또는 엑셀 임포트)
3. **스위트 정의**: Test Suite 탭에서 비즈니스 로직 단위로 케이스/플로우 묶음 생성 → 기본 스위트 지정
4. **스위트 활용**: 스위트 "불러오기" 클릭 → 케이스 관리 탭에 합산 필터 적용 + 자동 실행 탭 자동 프리셀렉트 (여러 스위트·히스토리 조합 가능)
5. **자동 실행**: 케이스 선택 + Base URL 지정 → 실행 → 결과 자동 기록 → Discord / Slack 알림
6. **이력 재활용**: 히스토리에서 이전 실행 불러오기 → 케이스 관리 / 자동실행 탭에 즉시 적용
7. **분석**: 실행 분석 탭에서 Pass율 추이 / 취약 케이스 파악 → 대시보드에서 프로젝트 간 비교
8. **변경 추적**: 케이스 변경 이력 탭에서 언제 무엇이 바뀌었는지 확인
9. **문서 작성**: 실행 완료 후 테스트결과서 보완 → 배포결과서 작성

## 불변성 원칙

실행이 완료된 결과(`case_results`, `flow_results`, `mgr_snapshot`)는 절대 수정되지 않습니다.
`label`만 편집 가능하며, 댓글은 추가만 허용됩니다.
케이스 변경 이력(`case_histories`)도 자동 기록 후 수정·삭제 없이 보관됩니다.
이로써 실행 이력과 케이스 변경 이력 모두 감사 가능성(Audit trail)을 보장합니다.
